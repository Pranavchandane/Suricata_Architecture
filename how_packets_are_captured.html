How Packets Are Captured

// from the wire → NIC → kernel → npcap → scapy → your code

01 — The Full Journey (7 Layers)
Physical Wire
Layer 1
ETHERNET CABLE / WIFI
Data travels as electrical signals (copper) or radio waves (WiFi). These are raw bits — 0s and 1s — flying through the wire at millions per second. No meaning yet, just voltage changes.
↓signals received by network card
NIC (Network Card)
Hardware
NETWORK INTERFACE CARD
The NIC converts electrical signals into bytes. It reads the Ethernet frame's destination MAC address. In normal mode, it discards frames not addressed to it. In promiscuous mode, it accepts all frames it sees — even ones meant for other machines.
↓frame passed up via DMA to kernel memory
Kernel Buffer
OS Ring Buffer
KERNEL SPACE
The NIC uses DMA (Direct Memory Access) to write the packet bytes directly into a ring buffer in kernel memory — without involving the CPU. The kernel keeps a circular queue of raw frames waiting to be processed. This is extremely fast.
↓Npcap (Windows) hooks here and makes a copy
Npcap / WinPcap
Windows Driver
KERNEL DRIVER (NDI FILTER)
Npcap installs as a kernel-level NDIS filter driver. It sits between the NIC driver and the Windows networking stack. When a packet arrives, Npcap makes a copy and puts it in a separate buffer accessible from user space. The original packet still flows to Windows networking normally — your machine keeps working.
↓copy passed to user space via wpcap.dll
libpcap API
User Space
USERSPACE LIBRARY
libpcap (or wpcap.dll on Windows) provides a standard C API to read those copied packets. It handles: opening the interface, setting promiscuous mode, applying BPF filters (Berkeley Packet Filter) to only capture specific traffic, and reading raw bytes packet by packet from the kernel buffer.
↓scapy wraps libpcap in Python
Scapy sniff()
Python
YOUR CODE
Scapy wraps the libpcap calls. sniff(iface=..., prn=callback, store=False) runs a loop: read next packet from buffer → parse headers → call your on_packet() function → repeat. store=False means packets aren't kept in memory — they're processed and discarded, keeping RAM usage low.
02 — Normal Mode vs Promiscuous Mode
🔒 Normal Mode
DEFAULT BEHAVIOR
NIC only accepts packets where the destination MAC = its own MAC (or broadcast). All other packets on the network segment are silently dropped at hardware level. You only see traffic meant for your machine.
🔓 Promiscuous Mode
IDS MODE
NIC accepts every frame it physically sees on the network segment — regardless of destination MAC. Your machine sees all traffic. This is what Npcap enables. Required for an IDS to monitor the whole network.
03 — What Happens When a Packet Arrives

TCP/IP STACK (normal flow)
Your browser, apps, OS networking all work normally. This path is untouched.
Copy → Npcap Buffer → Scapy
NPCAP NDIS FILTER (capture path)
A byte-for-byte copy goes into Npcap's buffer. Scapy reads it. Your IDS analyses it.
Key point: Packet capture is non-destructive — you're reading a copy. The original packet reaches its destination normally. Your IDS is just silently watching, like a CCTV camera on a road — traffic flows freely, but everything is recorded.
04 — Npcap → Scapy → Your Code
STEP 01
Npcap Opens Interface
Npcap driver opens the NIC, enables promiscuous mode, and starts copying incoming frames into a kernel ring buffer.

STEP 02
Scapy sniff() Loop
Scapy enters an infinite loop, reading one packet at a time from the Npcap buffer. Each packet is passed to your callback instantly.

STEP 03
Your Callback Called
For every packet captured, scapy calls your on_packet() function with a Scapy packet object containing all parsed layers.

05 — BPF Filters (Optional but Useful)
BPF = Berkeley Packet Filter — You can tell Npcap to only pass certain packets to your code, filtering at kernel level before they even reach Python. This reduces CPU load massively on busy networks.
